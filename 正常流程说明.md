## 一、系统核心发布流程（从飞书一句话到真正执行发布）

先按“**从简单到复杂**”梳理一条典型主流程，然后再扩展分支。

### 1. 典型主流程：创建一个新的发布计划并登记分支

场景示例（飞书里）：

> “帮我在这周五晚 8 点发 WMS 到 UAT，分支是 `feature/xxx`，仓库是 `wms-service`”

整体时序（高层）：

1. **飞书用户发送文本消息**
   
- 飞书 → 你的后端 `/bot/feishu/events`（消息事件回调）
   
2. **Bot 路由层解析请求**（`routers/bot_feishu.py`）
   - 提取：文本内容、用户 ID、会话 ID 等
   - 调用：`llm_service.parse_intent(text, context)` 得到 `IntentResult`
     - 例如：
       - `intent = "CREATE_RELEASE_PLAN_WITH_BRANCH"`
       - `params = { system_name: "WMS", environment: "uat", scheduled_time: "...", repo_name: "wms-service", branch_name: "feature/xxx" }`

3. **根据 Intent 分发到业务层**（仍在 `bot_feishu` 里）
   - `if intent == "CREATE_RELEASE_PLAN_WITH_BRANCH":`
     - 开启一个 DB session
     - 调用 `release_service.create_plan_with_single_branch(...)`

4. **业务服务：创建计划 & 分支登记**（`services/release_service.py`）
   - 如果是“带时间”的计划：
     - 创建 `ReleasePlan`（`status = "SCHEDULED"` or `"DRAFT"`）
   - 创建一个 `ReleaseItem`：
     - 记录仓库名、分支名（以及后续要补的 commit_sha）
   - 返回创建后的计划信息（含 ID）

5. **Bot 生成回复内容并返回给飞书**
   - 文本内容类似：
     - “好的，已为系统 WMS 创建 UAT 环境的发布计划（本周五 20:00），计划 ID: 123，已登记分支 wms-service:feature/xxx。”
   - 首版可以在 `bot_feishu` 里直接拼文案；
     如果想更智能，再调一次 LLM 让它根据 `IntentResult + DB 结果` 来润色回复。

6. **计划创建成功后，等待定时任务触发**
   - 发布计划表中有 `scheduled_time` 字段；
   - 系统里有一个 APScheduler 任务（例如每分钟跑一次）：
     - 在 `scheduler.py` 中：
       - `scan_and_trigger_scheduled_plans()`

7. **调度任务执行：扫描并触发到时间的计划**（`core/scheduler.py` + `services/release_service.py`）
   - 查询所有：
     - `status = "SCHEDULED"`
     - `scheduled_time <= now`
   - 对每个计划：
     - 将计划状态置为 `"RUNNING"`
     - 调用 `release_service.execute_plan(plan_id)` 开始实际发布流程

8. **实际发布流程**（`release_service.execute_plan`）
   对于每个 `ReleaseItem`（分支/仓库）：

   1. **确认 commit（可选）**
      - 如果没记录 `commit_sha`，可调用 GitLab API：
        - 查询该分支最新 commit；
        - 写回 DB 中。
   2. **触发 GitLab 合并（如需要）**
      - 调用 `gitlab_service.merge_branch_to_target(...)`
      - 可能会创建 MR 或直接合并，视你的策略（可配置）；
      - 更新 `ReleaseItem` 状态（如：`MERGING`, `MERGED`, `FAILED`）。
   3. **触发 Zadig 部署**
      - 调用 `zadig_service.trigger_deploy(plan, item)`；
      - Zadig 返回一个 Job/Task ID；
      - 存入 DB 以便后续跟踪。

   整个计划执行完：
   - 如果所有项成功：`ReleasePlan.status = "SUCCESS"`
   - 如果有失败：`ReleasePlan.status = "FAILED"`（并可记录失败原因）

9. **执行结果通知（可选）**
   - 在 plan 执行完成后：
     - 使用 `feishu_service` 给当初的发起人或群里发通知：
       - “计划 123 已执行成功，涉及仓库 xxx，详情见链接 ...”
   - 或者暴露一个 Admin UI 来查看执行结果。

### 2. 其它常见 Intent 分支

在 LLM 解析层面，可以定义一小撮“**核心 Intent**”，业务上也需要配套逻辑：

1. `CREATE_RELEASE_PLAN`
   - 只创建计划，不登记分支；
   - 用户后续可以再说“把某分支登记到某个计划”。

2. `REGISTER_BRANCH_TO_PLAN`
   - 用户话术：
     “把 `wms-service` 的 `feature/xxx` 加到这周五晚的那个 WMS 发版计划里”
   - 逻辑：
     - 根据时间+系统+环境或 plan_id 找到对应计划；
     - 创建 `ReleaseItem` 记录。

3. `QUERY_PLAN_STATUS`
   - 用户：
     “帮我查一下今晚 WMS 的发版状态”
   - 逻辑：
     - 查询最近/指定时间的 WMS 计划；
     - 返回计划状态、每个分支项状态；
     - 用 LLM/模板组装一段自然语言回复。

4. `CANCEL_PLAN`
   - 用户：
     “把今晚的 WMS 发版取消掉”
   - 逻辑：
     - 找到计划；
     - 如果尚未执行（`SCHEDULED`），改为 `CANCELED`，并不再触发实际流程。

5. `HELP` / `UNKNOWN`
   - 无法明确意图时，返回帮助提示或澄清问题。

下面这段你可以直接复制到 Cursor，作为“系统业务流程说明”，然后让它基于这个说明来帮你在 `services/`、`routers/`、`scheduler.py` 等处补业务逻辑。

你可以在工程 Scaffold 搞定之后，把这段贴进去，对它说“根据这个流程补全/重构当前代码”。

```text
You are helping me implement the core business flow of a "release management center" backend built with FastAPI, SQLAlchemy, and OpenAI.

High-level flow (very important):

1. A Feishu (Lark) user sends a free-form text message in a chat.
2. Feishu sends an event callback to our backend at `/bot/feishu/events`.
3. In `app/routers/bot_feishu.py`:
   - We parse the incoming event payload.
   - We extract the plain text and basic context (user id, chat id, etc.).
   - We call `llm_service.parse_intent(text, context)` to get an `IntentResult`:
     - `intent: str`
     - `params: IntentParams`
     - `need_clarification: bool`
     - `clarification_question: Optional[str]`
4. Based on `IntentResult.intent`, we dispatch to our domain service functions, mostly in `release_service.py`.
5. For example, for an intent like `CREATE_RELEASE_PLAN_WITH_BRANCH`:
   - We open a DB session.
   - We create a `ReleasePlan` row with fields such as:
     - system_name
     - environment
     - scheduled_time
     - status (e.g. "SCHEDULED" or "DRAFT")
   - We create a `ReleaseItem` row that belongs to this plan:
     - repo_name
     - branch_name
     - (optional) commit_sha
   - We commit and return the persisted entities.
6. The Feishu bot then returns a simple textual confirmation back to the user, such as:
   - "OK, I created a release plan for system WMS in UAT at Friday 20:00, plan ID: 123, with branch wms-service:feature/xxx."
7. Separately, we have a scheduler (using APScheduler) that periodically scans for scheduled release plans that should be executed:
   - In `core/scheduler.py`, we define a job like `scan_and_trigger_scheduled_plans` that runs every minute.
   - This job:
     - Opens a DB session.
     - Finds all `ReleasePlan` rows where:
       - status = "SCHEDULED"
       - scheduled_time <= now (UTC)
     - For each such plan, it:
       - Updates `ReleasePlan.status = "RUNNING"`.
       - Calls a function like `release_service.execute_plan(plan_id)`.

8. In `release_service.execute_plan(plan_id)`:
   - We load the plan and its `ReleaseItem` children from the DB.
   - For each `ReleaseItem`, we perform steps like:
     1) If `commit_sha` is empty, call GitLab to fetch the latest commit for the specified branch, and save it.
     2) Optionally, initiate a merge in GitLab (e.g. merge the branch into a target branch like `release` or `main`), using `gitlab_service`.
     3) Trigger a deployment in Zadig using `zadig_service`, and store any job/task ID in the DB for tracking.
   - We track individual `ReleaseItem` statuses (e.g. PENDING, MERGING, MERGED, DEPLOYING, SUCCESS, FAILED).
   - When all items finish:
     - If all succeed, set `ReleasePlan.status = "SUCCESS"`.
     - If any fail, set `ReleasePlan.status = "FAILED"` and record a reason.

9. Optionally, after a plan finishes (success or failure), we use `feishu_service` to send a notification back to the original requestor or group chat.

We also have other possible intents that we want to support in the same architecture:
- `CREATE_RELEASE_PLAN`: create a plan without any branch, only schedule and basic metadata.
- `REGISTER_BRANCH_TO_PLAN`: register a specific repo/branch to an existing plan, by plan_id or by matching (system + environment + time window).
- `QUERY_PLAN_STATUS`: query the status of a given plan or the latest plan for a system/environment.
- `CANCEL_PLAN`: cancel a scheduled plan that has not started yet.
- `HELP` or `UNKNOWN`: when the LLM cannot confidently parse the intent, we either show help or ask for clarification.

What I want you to do now:

   - Which functions should exist in `release_service.py` (e.g. `create_release_plan`, `create_plan_with_branch`, `execute_plan`, `register_branch_to_plan`, etc.).
   - Which routers and endpoints should call these functions (especially `admin_release.py` and `bot_feishu.py`).
   - How `core/scheduler.py` should be wired to periodically call `release_service.execute_due_plans()`.

2. Then, generate or refine the following:
   - In `services/release_service.py`:
     - A clear service API for:
       - Creating a release plan.

       - Registering a branch to an existing plan.
       - Executing all due plans (or a single plan by ID).
   - In `core/scheduler.py`:
     - APScheduler initialization and a job that periodically calls into `release_service` to execute due plans.
   - In `routers/bot_feishu.py`:
     - A dispatcher that maps `IntentResult.intent` to specific service calls (for at least the main two intents: `CREATE_RELEASE_PLAN_WITH_BRANCH` and `QUERY_PLAN_STATUS`).
     - Simple text responses for each case (we can later replace with richer Feishu messages).

Important:
```
